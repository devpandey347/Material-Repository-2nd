43 When piping a large readable stream (e.g., from fs.createReadStream) to the res object in an Express application for a client with a very slow network connection, what mechanism prevents the server's memory from being exhausted by buffering the entire file?

Understanding Request, Response and Server Objects Hard
A. Node.js streams' built-in backpressure mechanism, where res.write() returns false when the kernel buffer is full, causing the readable stream to pause.
B. The TCP window size mechanism, which automatically throttles the connection.
C. The server automatically compresses the data stream, reducing its memory footprint before sending.
D. The Express res.pipe() method internally chunks the data into small, memory-safe segments before sending.

Correct Answer: Node.js streams' built-in backpressure mechanism, where res.write() returns false when the kernel buffer is full, causing the readable stream to pause.

Explanation:

The core mechanism that handles this is backpressure, a fundamental feature of Node.js Streams. The response object (res) is a writable stream. When the client's network buffer (managed by the OS kernel) is full, calls to res.write() (which is used internally by pipe()) will return false. The .pipe() method automatically listens for this signal and pauses the readable stream (the file stream) from reading more data. Once the buffer has drained and is ready for more data, the writable stream emits a 'drain' event, and .pipe() resumes the readable stream. This prevents the server from reading the entire file into memory at once.

44 In a microservices architecture, an operation to create a customer order requires two distinct actions: 1) The Order Service creates an order record. 2) The Payment Service processes the payment. D. The Saga pattern, where the Order Service initiates a compensating transaction to cancel the order upon payment failure.

Correct Answer: The Saga pattern, where the Order Service initiates a compensating transaction to cancel the order upon payment failure.

Explanation:

The Saga pattern is designed to manage data consistency across multiple microservices without using distributed locks or a two-phase commit, which are complex and can reduce availability. In a Saga, a sequence of local transactions is executed. If any transaction fails, a series of compensating transactions are run to undo the impact of the preceding successful transactions. In this case, if the payment fails, the Saga would trigger a 'CancelOrder' command in the Order Service, effectively rolling back the operation.

51 In the context of the CAP theorem, if a Node.js-based microservices system prioritizes Availability (A) and Partition Tolerance (P), it must sacrifice strong Consistency (C). What is a common practical implication of this trade-off?

B. The system will return stale data from a service's local cache or replica during a network partition, achieving 'eventual consistency'.

D. The system must use a globally distributed, strongly consistent database like Google Spanner.

Correct Answer: The system will return stale data from a service's local cache or replica during a network partition, achieving 'eventual consistency'.

Explanation:

The CAP theorem states that a distributed system can only provide two of the following three guarantees: Consistency, Availability, and Partition Tolerance. The practical result is eventual consistency, where a service might respond with slightly outdated (stale) data during a partition, with the expectation that the data will become consistent across the system once the partition heals.
